## 공유
### 코드리뷰 경험 공유
팀 빌딩을 마치고 간단한 아이스브레이킹을 하며, 각자의 경험을 공유하는 시간을 가졌다. 당시에는 1주차 개인 미션의 구체적인 진행 방식이 가닥이 잡히지 않았고, 팀원간 코드 리뷰를 진행할 수도 있다는 생각이 들어서 코드리뷰 해왔던 경험들을 공유했다.

Ludo 스터디 매칭 플랫폼 팀 프로젝트를 7개월간 진행하며 코드리뷰를 해왔는데, PR을 남길 때와 리뷰를 남길 때 고려했던 부분들을 공유했다.

PR 남길 때
- 필요한 부분, 설명하고 싶은 부분을 코드를 첨부하여 설명하는 편
- 고민했던 부분을 PR 템플릿에 추가하면 좋겠다는 아쉬움이 있었음

리뷰 남길 때
- 코드 개선이 필요한 부분이 있으면 공감을 먼저 하고, 개선할 수 있는 부분의 코드 예시와 적용했을 때의 장점을 함께 제안
- 아는 부분이 있으면 공유 + 필요 시 코드 첨부

<br>

다른 팀원의 코드리뷰 경험도 공유받았고, 이를 종합하여 우리 팀에게 적용할 방법을 도출했다.
- 단계별 기능 목록
- 구조 설명 + 이렇게 구현한 이유
- 고민했던 부분

<br>

우테캠의 진행방식 상, 팀원간 코드리뷰에 조금 제약이 있어 진행하진 못했지만, 서로의 경험과 생각을 공유하는 것만으로도 의미가 있었던 시간이다.

<br> <br>

### 생산성 툴 공유
평소에 Excalidraw 그림 툴을 굉장히 애용한다. 간단한 그림을 그리기 굉장히 편리하고, 평소에 그림을 그리며 정리하는 것을 선호하는 편이라 많이 사용하는 편이다. 해당 툴을 활용했던 사례들과 함께 툴을 팀원들에게 소개해주었는데 반응이 굉장히(?) 폭발적이었고, 어느새 팀원 모두 그 툴을 사용하고 있었다😂. 앞으로도 이런 편의성 도구들이 있으면 적극적으로 공유할 예정이다.

## 기술적 고민과 생각
미션을 진행하며 고민했던 내용들을 간략히 기록하고자 한다.

### 정적 메서드 패턴
정적 메서드 시그니쳐는 상세할수록 그 의미를 명확히 전달할 수 있는 장점이 있다. 하지만 재사용성이 떨어지는 트레이드 오프가 있다.
```java
public static Piece createWhitePawn() {
    return new Piece(...);
}
public static Piece createBlackPawn() {
    return new Piece(...);
}
public static Piece createWhiteQueen() {
    return new Piece(...);
}
public static Piece createBlackKing() {
    return new Piece(...);
}
// ... 의미는 명료하지만, 너무 많아지는 정적 메서드
```

이럴 땐 정적 메서드의 네이밍과 파라미터를 적절히 조합하는 것을 고려해보면 좋다.
```java
public static Piece createWhiteOf(PieceType type) {
    return new Piece(WHITE, type);
}
public static Piece createBlackOf(PieceType type) {
    return new Piece(BLACK, type);
}
```

정적 메서드의 재사용성을 극한(?)으로 활용한 것이 일반적인 네이밍 컨벤션으로 활용되는 `of`와 `from`인데, 이러면 의미의 명료성이 떨어지는 단점이 있다. 지속적으로 리팩터링하며 적정 기준을 잡고 팀 내에서 컨벤션화 하는 것이 중요하겠다.

<br>

### Enum 사용 시점
상수간의 명확한 관계성을 띄고 있을 때 Enum을 사용하면 좋다. 또한 문자열에 대한 `Type-Safety`를 컴파일 타임에 확인하고 싶을 때 사용하면 좋다. 또한 개선된 Switch문과도 궁합이 좋으니 고려해보면 좋다.

<br>

### 추상 클래스 vs 인터페이스
이 부분은 정답은 딱히 없는 것 같고, 취향차이인 것 같다. 나는 개인적으로 추상 클래스를 선호하지 않는다. 정확히는 `extends` 상속을 별로 좋아하지 않는다. 상속으로 인해 부모 - 자식 클래스간 캡슐화가 완전히 깨지게 됨으로서, 기능 개발이 복잡해진 경우 기능 수정이 매우 어려워질 가능성이 높다고 생각한다. 부모의 변경이 자식에게 직접적인 영향을 끼치고, 마찬가지로 자식의 변경이 부모에게 직접적인 영향을 끼치기 때문이다.

반면 얻을 수 있는 장점은 코드의 재사용성인데, 이는 컴포지션을 활용하거나 인터페이스의 default 메서드를 활용하는 등 추상 클래스를 대체할 수 있는 부분들이 충분히 있다고 생각한다. 결론지으면, 단점에 비해 장점이 얻는 이점이 미미하고, 그 장점은 다른 기술적인 부분으로 어느정도 얻을 수 있기 때문에 추상 클래스를 선호하지 않는다.

## 팀원간의 피드백
한 주를 마치는 주간 회고 시간동안 팀원간의 피드백 시간을 가졌다. 

먼저 서로에게 잘한점을 칭찬하는 시간을 가졌다. 빠르게 개발을 마치는 다른 크루들을 보며, 목요일부터 속으로 의기소침해있었고 스스로에게 아쉬움이 많이 남아서 힘들었는데, 과분한 칭찬을 많이 해주셔서 부끄럽기도 하고 정말 감사했다.  가장 의외였고 기분 좋았던 칭찬은 표현하는게 깔끔하다는 칭찬이었다. Ludo 스터디 매칭 플랫폼을 진행하며, 팀원에게 대화의 핵심을 얘기해주는게 좋을 것 같다는 피드백을 받은적이 있었는데, 조금은 개선이 된 것 같아 기분이 좋았다. 그리고 정리를 잘한다는 이야기를 많이 들었는데, 이 부분은 장점이자 단점인 것 같다. 정리를 꼼꼼하게 하면 일을 추적 및 관리하는데 용이하지만, 그만큼 시간이 오래걸린다. 정리의 적정선을 찾아야할 것 같다.
정리하면, 평소에 잘 인지하지 못했던 `정리하는 습관`, `하고싶은 말은 깔끔하게`, `공유하는 습관` 강점들을 앞으로도 더 살려가볼 예정이다.

두 번째로는 서로에게 아쉬웠던 점을 피드백 남기는 시간을 가졌다. 일반적으로 사회 생활을 하다보면 서로에게 피드백을 주고 받을 수 있는 순간이 별로 없다. 하지만 성장을 위해선 이러한 소리들을 가끔씩은 들을 필요가 있고, 스스로도 들을 준비가 되어있어야 한다. 성장을 위해 모인 사람들이기에, 서로에게 피드백을 남기는 시간을 가지기로 했다.

고민을 너무 많이 하는건 내가 풀어야할 숙제다. 1주차 미션을 진행하며 생각이 너무 많아져서 개발 속도가 더디어졌는데, **개발의 0순위 우선순위는 마감일까지 기능 구현을 성공적으로 마치는 것**이다. 더 좋은 설계, 클린 코드는 그 이후의 얘기다. 이를 알고있음에도, 너무 많은 생각들을 적절히 쳐내지 못했던 부분은 앞으로 개선해야할 부분이다. 앞으로 개발을 진행하며 어디까지 고민하고 생각해야하는지 스스로의 기준을 정할 필요가 있다.

그리고 우테캠 1주차는 컨디션 조절을 못했다. 교육장까지 왕복 4시간이 걸리다보니, 체력 분배를 제대로 하지 못했다. 앞으로 루틴한 삶을 보내며 주어진 환경에 몸을 맞출 필요가 있다. 1차적인 목표는 바이오리듬을 우테캠에 맞추는 것, 2차 목표는 왕복 4시간의 시간을 활용하는 것이다. 이런 피드백을 남겨주신 팀원들에게 너무 감사하다.
