## 현상
로그인 API를 통해 **로그인에 성공**했음에도, 이후 다른 API를 호출하는 과정에서 **401 응답 발생 및 자동 로그아웃** 발생

(자동 로그아웃이 된 이유는 프론트엔드에서 서버로부터 401 응답을 받으면, 로그아웃 API를 호출하도록 구현했기 때문)

한 가지 신기한 점은 해당 현상이 ***간헐적으로*** 발생한다는 점

이 글은 해당 현상에 대한 트러블 슈팅에 대한 기록임. (편의를 위해 비격식체를 사용합니다)

<br><br>

## 이슈 추적
Ludo 서비스는 **`클라이언트 IP 주소`** 와 **`User Agent`** 정보를 바탕으로 세부적인 사용자 인증을 수행하고, 해당 세부 인증이 실패할 경우 `401 응답`을 내려준다. 따라서 `401 응답`의 원인을 추적하기 위해 **`클라이언트 IP 주소`** 와 **`User Agent`** 로그를 추적해 보았다.

사용자가 로그인에 성공할 때 저장하는 `Client IP / User Agent`와 다른 API를 호출할 때 요청으로 받은 `Client IP / User Agent` 정보 중 **`Client IP` 가 다른 것을 확인**할 수 있었다.

<img src="https://github.com/june-777/june-777.github.io/assets/68291395/7a5b36e3-d94b-472f-9e69-42c81630b15b">

<br>

각각의 **IP 주소가 2개**라는 점이 특이한 점이었고, **각각의 IP 주소가 가리키는 대상이 무엇인지 식별할 필요**가 있었다. Ludo 서비스의 시스템 아키텍처는 `Nginx (Reverse Proxy)` → `WAS (Spring)` → `DB (MySQL)` 구조였는데, WAS 레벨에서 확인할 수 있는 로그가 위에처럼 제한적이어서, **Nginx 레벨에서 Access 로그를 확인**해 보고자 했다.

확인한 결과, 위에 로그에서 확인할 수 있었던 `172.69.33.73`, `172.69.33.72` IP로부터 접근하고 있는 로그를 볼 수 있었다.

<img src="https://github.com/june-777/june-777.github.io/assets/68291395/e2a361cd-68ca-4cde-8fa8-57e3f38508f3">

<br>

**동일한 사용자가 동일한 환경에서 요청을 보내고 있는데, Nginx에서 추적한 IP 주소는 매 번 다르다는 점**이 이해가 되지 않았다.

Ludo 서비스의 시스템 아키텍처는 `Nginx (Reverse Proxy)` → `WAS (Spring)` → `DB (MySQL)` 라 하였는데, 그럼 더 앞단의 시스템 컴포넌트인 **DNS**가 문제인가? 라는 의문이 들었다.

또한 최근에 DNS 호스팅 서비스를 가비아에서 CloudFlare로 이전하였는데, 로그인이 자동 로그아웃 되는 현상도 비슷한 시점부터 발생했기 때문에 합리적인 의심이라는 생각이 들었다. 그래서 CloudFlare에 대해 조사해보기로 했다.

일반적으로 `클라이언트 - DNS` 의 상호작용은 `클라이언트`가 `DNS`로부터 IP 주소를 받고, **클라이언트가 IP에 해당하는 서버에 요청을 보내는 구조** 다. 하지만 CloudFlare는 **서버 요청을 직접 보내고, 그 과정에서 CDN을 활용** 한다고 한다. 중요한 점은 ***CDN을 분산하여 활용*** 한다는 점이다. [CloudFlare 공식 문서](https://www.cloudflare.com/ko-kr/learning/cdn/what-is-a-cdn/)

<br>

로그인을 성공했음에도, 이후 API 호출 시 **세부 인증이 `Client IP`로 인해 실패** 하는 현상이 ***간헐적*** 으로 발생는 점이 이해가 가기 시작했다.

CloudFlare는 클라이언트의 요청을 직접 서버로 보내는 과정에서 ***여러대의 CDN을 분산하여 활용하고 있기 때문에 매 요청마다 IP(CDN IP)가 달랐던 것*** 이다. 우연의 일치로 **동일한 CDN 서버를 거치게 되면, IP 주소가 같기 때문에 세부 인증이 성공**하는 것이고, **다른 CDN 서버를 거치게 되면 IP 주소가 다르기 때문에 세부 인증이 실패**했던 것이다.

이를 그림으로 표현하면 아래와 같다.

<img src="https://github.com/june-777/june-777.github.io/assets/68291395/00699185-c07b-473f-b989-9ae8db8cb2c9">

<br>

<img src="https://github.com/june-777/june-777.github.io/assets/68291395/ae0a712e-8f11-4333-b540-c7b8c1a6e9dc">

<br>

원인을 정확하게 추론했는지 파악하기 위해, `172.69.33.73`, `172.69.33.72` **IP 주소가 정말 CloudFlare의 IP 주소가 맞는지 확인**해 봤다. [CloudFlare 공식 문서](https://www.cloudflare.com/en-gb/ips/)를 확인해 보면 해당 서비스가 제공 중인 IP 주소의 범위를 확인할 수 있다.

<img width=300 src="https://github.com/june-777/june-777.github.io/assets/68291395/0d851a80-4eb3-41b8-bd3c-a006c5f5d77d">

`IP 주소: 172.64.0.0`, `서브넷 마스크: /13` 을 계산해보면, **`172.64.0.0` ~ `172.127.255.255`** 범위이고, `172.69.33.73`, `172.69.33.72` 는 범위 안에 포함되므로, CloudFlare의 IP 주소가 맞는 것을 알 수 있다.

<br><br>

## 마무리
해당 트러블 슈팅을 통해 다음의 경험을 얻고 느꼈다.

**시스템 아키텍처의 컴포넌트별로 로그를 추적하여 원인을 추정하는 경험**을 얻었다.

**네트워크에 대한 기본기의 중요성**을 다시 한번 느꼈다.

CDN을 활용할 때 발생할 수 있는 단점을 인지하게 되었다. (클라이언트에 대한 정확한 추적이 힘들어질 수 있다는 점)

어떤 DNS 호스팅 서비스를 사용함에 따라, 세부 인증과 같이 내부 구현이 바뀌게 될 수도 있다는 점을 느꼈다.

정리한 글을 정규 미팅 때 공유하여, 팀원들과 대안에 대해 논의할 예정이다.
