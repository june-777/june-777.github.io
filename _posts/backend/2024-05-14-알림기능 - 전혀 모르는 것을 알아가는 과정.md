## 알림 기능을 맡게 되며
3주간의 길고 길었던 `2차 마일스톤 - 기획 단계`를 마치고, `2차 마일스톤 - 개발 스프린트`가 본격적으로 시작됐다. 2차 마일스톤의 기능은 크게 **`알림 기능`**, **`출석 기능`**, **`신뢰도 기능`** 이었는데, 그 중 **`알림 기능`** 을 담당하기로 했다. 도전할 수 있는 요소들이 많아 보였기 때문에 기획 단계 때부터 굉장히 맡고 싶었던 기능이다. 다행히 백엔드 팀원분들 모두 어떤 기능을 담당해도 상관없다고 하셔서 내가 담당하게 되었고, 이번 기회에 여러 챌린지를 해보고 싶다.

<br>

## 전혀 모르는 도메인
알림은 내가 모르는 것이 너무 많은 도메인이었다. 일단 자료 조사를 해보고 내가 모르는 것들을 리스트업 해보기로 했다.

### 내가 모르는 것들 리스트 업

- Polling
- Long Polling
- Web Socket
- Server Sent Events
- FCM
- SendGrid
- MailGun
- SseEmitter
- 비동기 처리
- Spring AOP
- 이벤트 기반 아키텍처
- 메시징 큐

<br>

정리를 했으니 **분류를 해보자.** 대부분의 블로그를 보니 (Long) Polling, Web Socket, Server Sent Events와 FCM을 묶어서 비교하고 있었다. 여기서 의문이 조금 들었다. FCM은 메시징 매니지먼트 서비스 아닌가?

고민을 조금 더 해보니 **클라이언트에게 알림 이벤트를 전달하는 방법**을 기준으로 비교한 것으로 보였다. 전자의 4개의 경우 백엔드 서버에서 직접 구현하여 알림 이벤트를 응답으로 내려주는 것이고, 후자의 FCM의 경우 외부 서비스에게 알림 이벤트 전달까지 위임하는 구조가 되기 때문이다.



**클라이언트에게 알림 이벤트를 전달하는 방법**
- Polling
- Long Polling
- Web Socket
- Server Sent Events
- FCM

**분류 미정**
- SendGrid
- MailGun
- SseEmitter
- 비동기 처리
- Spring AOP
- 이벤트 기반 아키텍처
- 메시징 큐

<br>

아직 분류가 정해지지 않은 항목들이 많지만, 클라이언트에게 알림 이벤트를 전달하는 방법을 결정하는 것이 **업무 우선순위가 가장 높았다.** **어떤 방법을 선택하냐에 따라 백엔드 및 프론트엔드의 구현 로직이 결정**되기 때문이다. 따라서 정규 미팅 때 아래와 같이 팀원들께 제안했고, 다음 정규 미팅 때 다 같이 논의하여 결정하기로 했다.

<img width=500 src="https://github.com/june-777/june-777.github.io/assets/68291395/a12108ce-1c04-4e7a-aea9-e0f137892bc6">


Polling, Long Polling, Web Socket, Server Sent Events, FCM 에 대해 최우선적으로 알아볼 예정이고, 현재 서버 스펙(1 cpu, 3 mem)과 비즈니스 요구사항을 고려하여 결정할 예정이다. 이후에 아직 분류가 안된 것들에 대해 단계별로 알아보자.
